# 子 Agent 方案拆解技能

## 目标

把已有方案改写成可由主 agent 协调多个子 agent 执行的方案。只改写方案，不执行原方案。

输出必须让主 agent 直接看清：

- 哪些步骤由主 agent 自己完成。
- 哪些步骤必须或可以委派给子 agent。
- 哪些任务可并行，在哪里等待汇合。
- 哪些任务需要上下文隔离、专业审查或 evaluator agent。
- 每个子 agent 应拿到什么输入、返回什么输出、满足什么验收标准。

## 核心原则

- 保留原方案目标、约束和意图；不要新增用户未要求的新目标。
- 只有委派能带来上下文隔离、并行效率、专业审查、重复批处理或独立验证收益时，才使用子 agent。
- 主 agent 必须保留全局目标、关键取舍、冲突处理、方案集成、最终验收和用户可见输出。
- 子 agent 任务必须边界清晰、输入范围明确、输出格式明确、可独立验收。
- 委派收益不明确时，优先标记 `[[SUBAGENT_OPTIONAL]]` 或保留给主 agent。
- 不要为了显得复杂而增加 agent；协调成本必须低于委派收益。

## 标记定义

| 标记 | 使用规则 |
| --- | --- |
| `[[MAIN_AGENT_ONLY]]` | 目标定义、全局约束、关键决策、冲突合并、最终验收、最终用户可见输出，或任何必须依赖完整全局上下文的步骤。 |
| `[[SUBAGENT_REQUIRED]]` | 必须委派。用于上下文量大、边界清晰、专业性强、重复批处理、多方案探索或独立验证收益明确的步骤。 |
| `[[SUBAGENT_OPTIONAL]]` | 可以委派但非必须。用于复杂度中等、收益存在但主 agent 也可直接处理的步骤。 |
| `[[SPLIT_TO_SUBAGENTS]]` | 一个原步骤应拆成多个独立工作包，例如按文档、模块、方案、风险类别、用户群或数据源拆分。 |
| `[[PARALLEL_GROUP: <id>]]` | 同组任务可并行执行。同一组内任务不能互相依赖。 |
| `[[JOIN_GATE]]` | 主 agent 必须等待多个输出返回后再继续，并负责合并、冲突处理和 go/no-go 判断。 |
| `[[DEPENDENCY: <id>]]` | 当前步骤依赖某个前置步骤、子 agent 输出或 join gate。 |
| `[[CONTEXT_DELEGATE]]` | 委派原因是上下文隔离。子 agent 必须返回压缩后的结构化结论，不能搬运长篇原文。 |
| `[[REVIEW_AGENT]]` | 需要独立审查、风险检查、安全/合规检查、测试设计、逻辑完整性检查或 evaluator-optimizer 式批判。 |

可追加原因标签，例如：`reason=可并行`、`reason=上下文过大`、`reason=需要专业视角`、`reason=重复批处理`、`reason=多方案探索`、`reason=需要独立审查`、`reason=合并风险过高`、`reason=主 agent 直接处理成本更低`。

## 委派判断

按顺序判断每个原子步骤：

1. 如果该步骤定义目标、约束、优先级、最终判断或用户可见结论，标记 `[[MAIN_AGENT_ONLY]]`。
2. 如果该步骤需要阅读大量材料，而主 agent 只需要摘要或结构化结果，标记 `[[SUBAGENT_REQUIRED]] [[CONTEXT_DELEGATE]]`。
3. 如果该步骤可拆成多个互不依赖的对象或方向，标记 `[[SPLIT_TO_SUBAGENTS]]`，并为拆出的任务设置 `[[PARALLEL_GROUP: <id>]]`。
4. 如果该步骤需要专业审查或独立批判，标记 `[[REVIEW_AGENT]] [[SUBAGENT_REQUIRED]]`。
5. 如果该步骤是大量重复处理，按对象、批次、来源或主题拆分给多个子 agent。
6. 如果该步骤需要探索多个备选方案，可让多个子 agent 独立评估不同方案，再由主 agent 合并排序。
7. 如果任务边界、输入范围、输出格式或验收标准写不清，不要委派；先由主 agent 澄清或自行处理。

不应委派的情况：任务很小、强依赖完整上下文、价值判断不可外包、前后步骤高度耦合、结果缺少验收标准、合并成本高于收益。

## 处理流程

1. 标准化原方案：拆成原子步骤，保留原意，不新增目标。
2. 识别依赖：为每一步列出输入、输出、后续依赖、可提前执行性和可并行性。
3. 分类步骤：为每一步添加主 agent、子 agent、并行、上下文隔离、审查和依赖标记。
4. 创建并行组：把互不依赖的任务放入同一 `[[PARALLEL_GROUP: <id>]]`。
5. 添加 join gate：每个并行组或多 agent 拆分后必须有 `[[JOIN_GATE]]`。
6. 编写子 agent 任务简报：为每个委派任务定义角色、目标、输入、不要做什么、输出格式、验收标准、依赖项和压缩等级。
7. 添加审查步骤：仅在复杂、高风险、影响面大或关键决策场景添加 review agent。
8. 自检：确认委派收益大于协调成本，依赖关系闭合，最终责任仍在主 agent。

## 输出结构

最终输出必须包含以下部分，且顺序固定：

1. `拆解总览`
2. `依赖与并行关系图`
3. `改写后的 Subagent-Ready 方案`
4. `子 Agent 任务简报`
5. `汇合 / 合并协议`
6. `主 Agent 检查清单`

## 子 Agent 任务简报模板

```markdown
# 子 Agent 任务简报

**输入范围：** 列出可使用的文件、文档、章节、数据源、约束或上下文。

**不要做：** 说明不得执行的动作，尤其是哪些决策保留给主 agent。

**输出格式：**

- 关键发现：
- 证据：
- 风险：
- 建议：
- 未解决问题：
- 置信度：
- 给主 agent 的集成提示：

**验收标准：**

- 输出简洁且可合并。
- 包含必要证据或引用。
- 不搬运无关原始上下文。
- 明确标注不确定性。
- 不越权做最终决策。

**压缩等级：** 高压缩 / 中压缩 / 低压缩，并说明选择原因。
```

## 汇合 / 合并协议

每个 `[[JOIN_GATE]]` 必须说明：

- 必须等待哪些子 agent 输出。
- 主 agent 如何合并，而不是简单拼接。
- 输出之间冲突时如何处理。
- 采用什么证据优先级。
- 哪些条件满足后才能进入下一阶段。
- 哪些未解决问题必须显式保留。

默认规则：优先采用有证据支持、符合原目标和约束的结论；对证据不足的结论标记不确定性；少数派意见如果涉及重大风险，必须保留给主 agent 判断。

## 主 Agent 检查清单

最终输出前检查：

- 是否保留原方案目标，没有新增用户未要求的新目标。
- 每个子 agent 任务是否有明确输入范围、输出格式和验收标准。
- 每个并行组中的任务是否真的互不依赖。
- 每个并行组或多 agent 拆分是否都有 join gate。
- 依赖关系是否闭合，后续步骤是否引用了正确的 dependency id。
- 上下文重任务是否被压缩处理，而不是原文搬运。
- review agent 是否只在有收益时添加，且审查结果被处理或明确延后。
- 主 agent 是否保留关键决策权、冲突处理权和最终输出责任。
- 最终方案是否比“全部交给一个 agent”更容易执行。

## 质量要求

- 输出要精确、结构化、可执行。
- 不写空泛建议，不使用“分析一下”这类模糊任务。
- 不让子 agent 越权做最终决策。
- 不机械复制原方案再加标记。
- 不强行增加 agent；复杂度必须服务于执行效率。
